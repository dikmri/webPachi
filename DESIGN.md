# webPachi 設計書

GitHub Pages で遊べる物理演算パチンコ台シミュレーター。
機種はオリジナルの海テーマ甘デジ「P ドリームオーシャン 99ver.」
(スペック・釘構成は一般的な羽根なしデジパチ/甘デジの典型構成を参考にしたオリジナル)。

## 技術スタック

- bun + Vite + TypeScript (strict) + matter-js (物理)
- 描画: Canvas 2D (盤面) + DOM/CSS (液晶・データカウンター・枠)
- 音: WebAudio API による手続き生成のみ (外部アセット・既存楽曲は一切使わない)
- デプロイ: GitHub Actions → GitHub Pages

## イベントフロー

```
[board] --BoardEvent--> [main] --転送--> [game(GameLogic)]
[game]  --GameEvent---> [main] --配線--> [reels] [audio] [dataCounter] [board(役物開閉)]
[frame] --handlePower-> [main] --launch--> [board]
[main] が PlayerState/Stats を所有し賞球加算・差玉サンプリングを行う
```

- 賞球テーブルは `SPEC.payout`。main が BoardEvent を見て持ち玉に加算する。
- `GameLogic.attackerShouldOpen` / `denSupport` を main が毎フレーム見て
  `board.setAttackerOpen()` / `setDenchuOpen()` を呼ぶ (電チューはゲート通過で1.5s開放、
  電サポ中のみ。開放制御は board 側でなく main 側タイマーでよい)。

## 盤面仕様 (エージェントA)

論理座標 480x660。重力は matter-js デフォルト方向(下向き)。

- 玉: 半径 5.5、restitution≒0.4、friction 小、密度デフォルト。上限 60個(超えたら古い玉を消す)。
  最大速度を保険として px/秒でクランプ(異常な跳ね返りによる速度増幅対策)。
- 釘: 半径 2.0 の静的円、restitution≒0.42。全体で約140本(縁釘38+千鳥格子51+天釘3+
  命釘2+袖釘6+寄せ釘16+道釘左右26+ジャンプ釘2)。
- 外枠: 左右壁・上部は円弧(中心 (240,330) 付近、半径≒310 の上半分)。
- 発射レール: 左端に幅約14の誘導路(外壁と内レールの間)。玉は左下 (18,640) 付近から
  power に応じた初速で上方へ発射され、円弧に沿って盤面上部へ入る。
  強いほど右奥(ブッコミ超え)まで飛ぶ。レール出口に戻り防止片。
- センター役物(液晶枠): 矩形 x:130..350, y:150..330 の静的ボディ。
  左横にワープ入口(通った玉はステージへ)。ステージ=液晶下の皿(y≒335)で玉が左右に転がり、
  中央から落ちるとヘソ上に落下(ヘソ入賞率高)。
- 風車(ふうしゃ): 左 (105,290)・右 (375,290)。回転する十字体(ピン留め)。玉を散らす。
- 縁釘(外周に沿った釘列): 外壁(上部円弧の左肩〜解放点〜右肩〜右辺)のすぐ内側
  (オフセット15px)に弧長26px間隔で一列に並ぶ。実機の「誘導レールのすぐ内側に
  ずっと並ぶ釘列」を再現し、強めのハンドルで右壁沿いを転がる右打ちルートの
  土台にもなる。機能釘(命釘・道釘・袖釘・寄せ釘等)に近すぎるものは自動的に間引く。
- 天釘: (216,110)(240,106)(264,110) の3本。レール解放点直下で玉を最初に振り分ける。
- 千鳥格子(上部〜中段の密な釘field): y:122〜334 の範囲に行間隔24px・列間隔26pxの
  スタッガード配置で機械的に生成(中央液晶・風車の可動域は除外)。実機らしい
  「規則正しいが経路で微妙に振り分けが変わる」密な釘盤を再現する。
- 袖釘(センター役物まわりの「ハの字」漏斗): 液晶枠の左下・右下のすぐ外側に
  各3本、上ほど役物に近く下へ行くほど外側へ開く配置。抜けた玉を下段の
  寄せ釘へ引き継ぐ。
- 寄せ釘: 袖釘の下、y:408〜510 に左右各8本。下へ行くほど中央(ヘソ)側へ収束し、
  道釘・ジャンプ釘・命釘へ滑らかに繋がる「漏斗」を構成する。
- 道釘: y≒546〜566 に左右対称の水平列(間隔約13〜14、玉が上を転がる)。
  左列 x:40〜224・右列 x:256〜440。各列途中1箇所こぼしの切れ目。
  レール解放点が power で広く散らばるため、右側にも同型の道釘列を用意して
  片側だけに回転率が偏らないようにしている。
- ジャンプ釘→命釘: 道釘の終端で玉を跳ねさせ、命釘2本(中心間隔≒70)の間へ。
- ヘソ(スタートチャッカー): (240,558) 幅11のセンサー。
- 電チュー: ヘソ直下 (240,588)。閉時は入賞不可、開時は幅22の受け皿。
- スルーゲート: 左道中 (100,340)。通過検知のみ。
- アタッカー: (240,612) 幅56の開閉フラップ。開時は落下玉の大半を吸い込む。
- 一般入賞口: 左下に3個 (60,600)(85,614)(110,624) 付近。
- アウト口: 最下部 (240,648)。入賞しなかった玉を回収。
- 盤面背景: 海テーマ(青グラデ・泡・光条・砂地)を Canvas で手続き描画。「それっぽく」豪華に。

**貫通(tunneling)対策**: 固定ステップを 1000/240 秒に細分化、
positionIterations/velocityIterations を既定値より引き上げ、玉の最大速度を
保険としてクランプした上で、玉と釘専用の簡易連続衝突判定(CCDスイープ)を実装。
Matter.Engine.update() 前後の玉の座標を線分とみなし、近傍の釘(空間分割で
絞り込み)に対して「線分と釘中心の最短距離」が当たり判定幅を下回っていないか
幾何学的に判定し、両端点が当たり判定の外側なのに軌跡だけが貫通している場合
(=matter-js の離散衝突では検出されない tunneling)のみ接触点まで押し戻して
反射させる。scripts/simulate.ts に検証用カウンタ(検出・補正回数/補正漏れ回数)
を実装し、強めのpower(0.7〜1.0)で数千発のストレステストを行い
補正漏れ0件を確認済み。

**発射口への戻り防止**: レール内壁(`RAIL_INNER_WALL`)は「昇降中の玉」向けには
物理ボディを作らない(isSensorのスクリプト移動なので何とも衝突しない)が、
放出後の通常の玉が盤面左下をバウンドして発射口の通り道へ戻り込んでしまう
バグがあったため、レール直線区間(`RAIL_CENTERLINE`の最初の13点、発射口
付近〜円弧が始まる直前、おおよそy:106〜668)だけを物理ボディ化した。
**円弧区間(玉の解放点を含む上部の弧)は絶対に含めないこと**: 一度試したところ、
解放直後の玉が自分の来た壁(解放点=弧の終端)に接線方向で干渉し、全power帯で
ヘソ入賞が0発になる深刻な回転率崩壊を実測で確認している。直線区間のみの
物理化であればこの問題は起きず、`bun run simulate` の数値・貫通対策の
escaped=0も維持されることを確認済み。

**調整目標(scripts/simulate.ts で検証)**: 適正ハンドル(power 0.55〜0.65 付近)で
1000発打って ヘソ入賞 55〜75発 (千円=250発あたり14〜19回転)。
シミュレーションは matter-js を固定タイムステップで回し、入賞内訳を表で出力する。
(シミュレーションは発射ごとに乱数ジッターを与えているため実行のたびに数発程度の
ばらつきがある。目安から外れる回が稀にあっても、複数回実行して大勢として
55〜75発付近に収まっていれば正常。)

## 盤面エディタ (エージェントA)

盤面の釘・役物配置を「人力で」調整したいという要望に応え、ブラウザ上で
座標を直接編集できるツールを用意した。`editor.html` を開くと使える
(`index.html` とは完全に独立した別ページ)。

### 「筐体」と「取付部品」の切り分け

実機になぞらえ、盤面を2種類の要素に分けている。

- **筐体(編集不可・固定)**: 外枠レール(`RAIL_*`)・右側の外枠
  (`FIELD_RIGHT_WALL`)・アウト口(`OUT_ZONE`)。これらは `src/board/layout.ts`
  に定数として残り、エディタでは編集できない。
- **取付部品(編集可能)**: 釘・ヘソ・電チュー・スルーゲート・アタッカー・
  一般入賞口・風車の**座標**、および**センター役物(液晶枠)の位置**。
  実機の技術者が釘配置やチューリップ位置をハンマーで調整するのと同じ
  発想で、これらは `BoardData` という単純なデータ型として外部化し
  (`src/board/boardData.ts`)、エディタから自由に編集・保存できるように
  した。釘の個別半径やヘソ等の半幅も編集できるが、釘の既定半径
  (NAIL_RADIUS)・一般入賞口の半幅(POCKET_HALF_WIDTH)・風車の腕の寸法や
  回転速度・**センター役物のサイズ(幅220×高さ180)**など「サイズ・挙動」は
  引き続き固定。

「モニタの位置も調整したい」という追加要望に応え、センター役物
(`BoardData.centerBox`。中心座標のみ)をドラッグで動かせるようにした。
ステージ(`stageLeftFor`/`stageRightFor`)・ワープ入口(`warpEntranceFor`)・
ワープ転送先(`warpTargetFor`)は、すべて `centerBoxRectFor(data)` から
算出する関数として `src/board/layout.ts` に実装されており、センター役物を
動かすとこれらも自動的に追従する(個別に動かす必要はない)。

物理演算(`PhysicsCore`)・描画(`drawBoard`)はどちらも `BoardData` を
引数として受け取るように統一されているため、本編(`src/main.ts`)は
`new Board(data)` に読み込んだ `BoardData` を渡すだけでよく、エディタで
組んだ盤面がそのまま実機と同じ物理エンジンで動く(挙動の二重実装がない)。

### 新障害物: バー・回転体・パスカーブ

「釘だけでなく、バー・回転体・パスカーブによる障害物も置けるようにして、
もっと盤面をコントロールできるようにしたい」という要望に応え、釘・役物とは
別枠の「自由配置できる障害物」として3種類を追加した(`BoardData.bars` /
`BoardData.spinners` / `BoardData.curves`)。いずれもデフォルト盤面
(`DEFAULT_BOARD_DATA`)では空配列であり、既存の物理・描画・回転率
シミュレーション結果には一切影響しない(`bun run simulate` の数値は
従来と完全に同一)。

- **バー(`BarObstacle`)**: `x1,y1`〜`x2,y2` の間に厚み `thickness` の
  静的な壁を作る単純な直線障害物。`layout.ts` の `buildBarBody()` が
  既存の `buildSegment()`(ステージ等で使用)を反発係数指定つきで呼ぶ
  だけの薄いラッパーで、ラベルは `"obstacle-bar"`(既存の
  `"nail"`/`"wall"`/`"stage"` とは衝突しないだけの、ただの静的障害物。
  `handleCollisions` 側で特別扱いは不要)。反発係数(`restitution`)は
  省略可能で、省略時は釘と同じ既定値を使う。
- **回転体(`SpinnerObstacle`)**: 中心 `x,y` を軸に一定角速度
  (`spinSpeed`、ラジアン/秒)で回転する単腕の回転バー。既存の風車
  (`BoardData.windmills`。十字型・固定形状・寸法固定)とは全く別物で、
  位置・長さ・太さ・回転速度をすべて自由に設定できる。`layout.ts` の
  `createWorld()` 内で風車(`WindmillState`/`tickWindmills()`)と全く同じ
  パターン(アンカーにピン留めしつつ`Matter.Body.setAngularVelocity`で
  回転させる)で `SpinnerState[]`/`tickSpinners()` を実装している。
  現在の回転角は `PhysicsSnapshot.spinnerAngles` 経由で描画側に渡る。
- **パスカーブ(`CurveObstacle`)**: `points`(3点以上・奇数個)による
  区分2次ベジェの曲がった壁。連続する3点ずつ(始点・制御点・終点)が
  1区間を構成し、区間同士は端点を共有して繋がる。`layout.ts` の
  `buildCurveBodies()` が既存の `sampleBezier()` で各区間を細かく
  サンプリングし、既存の外壁構築(`buildWallChain()`)と同じ手法で
  連続する小さな回転矩形を繋いだ静的ボディ列にする。エディタの配置
  ツールは3点(始点・制御点・終点)を1個作るだけの MVP だが、型・
  物理構築とも5点以上の連続スプラインを見越した一般形になっている。

いずれも `handleCollisions` 側の特別なゲート処理は不要(玉が跳ね返る
だけの純粋な静的障害物)。旧形式データ(`bars`/`spinners`/`curves`
フィールドが無い localStorage / JSON)は、`centerBox` の救済と全く同じ
パターンで `normalizeBoardData()` が空配列を補う(詳細は次項)。

### データの保存の仕組み

- 編集するたびに(デバウンスして自動保存)、`localStorage` の
  キー `webpachi-board-data-v1`(`BOARD_DATA_STORAGE_KEY` として
  `src/board/boardData.ts` がエクスポート)へ現在の `BoardData` を
  JSON 保存する。
- 本編(`src/main.ts`)は起動時にこのキーを読み、`isValidBoardData()` で
  検証した上で問題なければその盤面を、無ければ `DEFAULT_BOARD_DATA`
  (従来通りの盤面)を使う。**つまりエディタで調整した盤面は、保存すると
  そのまま本編のプレイ画面にも反映される。**
- 「JSONをエクスポート」で `.json` ファイルとしてダウンロードでき、
  「JSONを読み込む」でファイルから復元できる(どちらも
  `isValidBoardData()` で形状を検証し、不正なら画面にエラー表示する)。
- 「デフォルトに戻す」で `DEFAULT_BOARD_DATA` のコピーに戻せる
  (localStorage も上書きされる)。
- 「釘をすべて消す(プレーン化)」で `data.nails` だけを空配列にできる
  (「デフォルトに戻す」とは別物で、ヘソ・電チュー・ゲート・アタッカー・
  一般入賞口・風車・センター役物の位置はそのまま維持される)。Undo可能
  なので確認ダイアログは出さない。
- **旧形式データの救済**: `BoardData.centerBox`(センター役物の中心座標)を
  追加した際、既に localStorage / JSON に保存済みの編集データには
  `centerBox` フィールドが存在しない。`isValidBoardData()` は
  `centerBox` の有無を必須にしておらず(無くても他の必須フィールドが
  揃っていれば通過する)、代わりに `normalizeBoardData(v)` が
  `centerBox` 欠落時に `DEFAULT_BOARD_DATA.centerBox` のコピーを補って
  返す。localStorage / JSONファイルから読み込んだ直後は、エディタ
  (`src/editor/main.ts`)・本編(`src/main.ts`)のどちらも必ず
  `normalizeBoardData()` を通してから使うことで、旧形式データを
  読み込んでもユーザーの編集内容が失われず、センター役物だけ従来通りの
  中央位置で正常に動作する。バー・回転体・パスカーブ
  (`bars`/`spinners`/`curves`)を追加した際も全く同じパターンで対応した。
  これらのフィールドが無い旧データも `isValidBoardData()` を通過し、
  `normalizeBoardData()` が欠けている分だけ空配列 `[]` を補う
  (ユーザーの釘・役物・新障害物の編集内容は一切失われない)。

### エディタの操作方法

- **釘の追加**: キャンバスの空いている場所をクリック。
- **釘・役物の移動**: 既存の釘・役物(ヘソ/電チュー/ゲート/アタッカー/
  一般入賞口/風車/センター役物)をドラッグ。役物は移動のみ可能で削除は
  できない。釘は選択して `Delete`/`Backspace` キーで削除できる。
- **新規釘の半径**: ツールバー「釘の配置」パネルの数値入力(デフォルトは
  `NAIL_RADIUS`)で、これから配置する釘の半径をあらかじめ調整できる。
  通常クリックでの釘追加・下記「直線配置」での一括配置のどちらにも
  適用される。あくまで「次に置く釘」への設定であり、既に配置済みの釘の
  半径は変わらない(既存釘の半径変更は引き続きプロパティパネルの
  「半径」欄で個別に行う)。
- **釘の直線配置**: ツールバー「釘の配置」パネルの「直線配置」ボタンで
  トグルON/OFFできる専用モード。ONの間は通常のクリック=釘追加ではなく、
  1回目のクリック(ドラッグなしのpointerdown→pointerup)で始点を確定し、
  マウスを動かすと始点〜現在のカーソル位置を結ぶプレビュー線(と実際に
  置かれる釘の位置の先読み表示)が出る。2回目のクリックで終点を確定すると、
  その瞬間に始点〜終点の間へ「配置間隔(px)」欄で指定した間隔に近い
  等間隔で釘をまとめて配置する(本数は
  `Math.max(2, Math.round(距離/間隔) + 1)`、始点・終点にちょうど乗るよう
  均等配分)。配置される各釘の半径は上記「新規釘の半径」設定に従う。
  一連の配置操作全体で Undo は1回分にまとめてあるので、まとめて
  取り消せる。モード中に `Esc` を押すと確定前の始点をキャンセルできる。
  グリッドスナップが有効な場合は始点・終点もスナップする。
- **センター役物(モニタ)の移動**: 220×180の大きな矩形のため、他の役物と
  同じ「中心からの円形距離」判定だと隅をクリックしても選択できず使いにくい。
  そのためクリック判定は2段階になっている: まず釘・ヘソ・電チュー・
  ゲート・アタッカー・一般入賞口・風車を通常通り最近傍判定し、**どれにも
  ヒットしなかった場合のみ**、クリック座標がセンター役物の矩形内かどうかを
  別途チェックする(優先順位を一番低くすることで、役物の上に置かれた釘
  などを誤って掴んでしまうのを防ぐ)。選択中は矩形の輪郭がハイライト
  表示される。ステージ・ワープもセンター役物の位置に自動追従するため
  個別に動かす操作は不要。
- **障害物(バー・回転体・パスカーブ)の配置**: ツールバー「障害物」パネルに
  釘の「直線配置」と同じ操作パターン(トグルON/OFF + クリックで座標確定)の
  専用モードが3つ並ぶ。同時に有効にできるのは1つだけで、いずれかを
  ONにすると他の配置モード(直線配置含む)は自動的にOFFになる。
  - **バー配置**: ONの間、1回目のクリックで始点、2回目のクリックで終点を
    確定し、その瞬間に `BarObstacle` を1個追加する。厚み・反発係数は
    パネルの数値入力(デフォルト厚み6px、反発係数は空欄=既定値)で
    事前設定できる。確定前はプレビュー線が表示される。
  - **回転体配置**: ONの間、1回のクリックで即座に `SpinnerObstacle` を
    1個作る(位置=クリック地点)。長さ・太さ・回転速度はパネルの数値入力
    (デフォルト長さ40px・太さ8px・回転速度1.5rad/s)に従う。
  - **カーブ配置**: ONの間、1回目=始点、2回目=制御点、3回目=終点の
    3クリックで `CurveObstacle`(3点)を1個作る。1・2回目クリック後は
    プレビュー曲線(2次ベジェ)が表示される。厚み・反発係数はバー配置と
    同様にパネルの数値入力で事前設定する。
  - いずれのモードも `Esc` で確定前の点をキャンセルでき、配置後は選択
    状態になってプロパティパネルで細かく編集できる。
- **障害物の移動・編集**: バーは両端点(始点/終点)を個別にドラッグして
  形を変えられるほか、中点をドラッグするとバー全体を平行移動できる
  (中点ドラッグはドラッグ先座標と現在の中点との差分ベクトルを計算し、
  その差分を両端点に加算する形で実装しており、バーの向き・長さは保たれる)。
  回転体は中心1点をドラッグで移動できる(既存のヘソ等と同じ扱い)。
  カーブは3つの制御点をそれぞれ個別にドラッグできる。いずれも釘と同じく
  選択して `Delete`/`Backspace` キーで削除できる(役物のような削除不可の
  制約はない)。選択中はバーなら線+3ハンドル、カーブなら曲線+3制御点が
  ハイライト表示される。
- 選択中のアイテムは右側ではなく左側の「プロパティ」パネルに表示され、
  x/y(・釘なら半径、役物なら半幅)を数値入力でも編集できる。バーを
  選択した場合は x1/y1/x2/y2・厚み・反発係数、回転体を選択した場合は
  x/y・長さ・太さ・回転速度、カーブを選択した場合は選択中の制御点の
  x/y・カーブ全体の厚みを、それぞれ専用欄で編集できる。
- 「表示切り替え」パネルのトグルで、
  - 釘の物理半径(実際の釘の太さ)の円
  - 釘の実効半径(釘半径+玉半径 `BALL_RADIUS`)の破線円 ―
    **2本の釘の間を玉がすり抜けられるかどうかの実際の目安**であり
    パチンコ盤面設計上もっとも重要な可視化(デフォルトON)
  - テスト発射中の玉自身の当たり判定円
  - グリッドスナップ(2px単位)
  を表示・切り替えできる。
- 「テスト発射」パネルの power スライダーと「1発発射」ボタン、または
  「オート連射」チェックボックス(ハンドルを握りっぱなしにした状態を再現)で、
  現在編集中の盤面がそのまま `PhysicsCore` に反映された状態で実際に玉を
  転がして確認できる(編集のたびに物理ワールドを再構築するため、
  常に最新の釘配置で動く)。
- 「簡易シミュレーション」パネルで power と発射数を指定して「実行」を押すと、
  `scripts/simulate.ts` と同じロジックをブラウザ内で(専用の使い捨て
  `PhysicsCore` を使って)ヘッドレス実行し、ヘソ/電チュー/ゲート/一般入賞/
  アタッカー/アウトの内訳とヘソ率・回転数の目安をその場に数値表示する。
  CLI を使わずにブラウザだけで回転率チューニングが完結する。
- 右側の「動作ログ」パネルには操作(釘追加・移動・削除・保存・読込・
  シミュレーション結果等)が逐次記録され、「ログをダウンロード」で
  テキストファイルとして保存できる。
- `Ctrl+Z` または「元に戻す」ボタンで直前の編集を1手戻せる(Undo、最大30手)。

### テスト発射への本編ゲームロジック統合(保留・変動・大当たり)

以前のテスト発射・簡易シミュレーションは `PhysicsCore`(生の物理)だけで
動いており、`PachinkoGame`(エージェントB担当、保留・変動・大当たり・
確変/時短)を一切経由していなかった。そのためエディタでヘソに玉を
入れても保留が増えず、変動も大当たりも起きないという実際のプレイ感と
乖離した挙動になっていた。これを解消するため、`src/editor/main.ts` は
`core`(PhysicsCore)と並べて `game`(`PachinkoGame` のインスタンス)を持ち、
`src/main.ts` の `tickLaunch`/`handleBoardEvent`/`tickDenchu`/
`openDenchuIfSupported`/`handleGameEvent` の `round-start`/`round-end`/
`jackpot-end` と全く同じ配線パターンをフレームループ内に再現している。

- 毎フレーム、`core.update(dtMs)` が返す `BoardEvent[]` を `game.onBoardEvent(ev)`
  に渡す。これによりヘソ入賞で保留が増え、保留があれば変動が始まり、
  当たれば大当たりのラウンド進行まで実際に起きるようになった。
- `gate`(スルーゲート通過)イベントが来たとき `game.denSupport` が true
  (確変または時短中)なら、本編と同じ `DENCHU_OPEN_MS`(1500ms)だけ
  `core.setDenchuOpen(true)` する専用タイマーをエディタ内に実装した。
- `game.update(dtMs)` が返す `GameEvent[]` をループし、`round-start` で
  `core.setAttackerOpen(true)`、`round-end`/`jackpot-end` で
  `core.setAttackerOpen(false)` を呼ぶ。加えて毎フレーム
  `core.setAttackerOpen(game.attackerShouldOpen)` で整合性を取る
  (本編と全く同じ二重の同期パターン)。
- `spin-start` イベントのたびにこのセッション限定のカウンタ
  `editorTotalSpins` を、`jackpot-start` イベントのたびに `editorJackpots`
  をインクリメントする。どちらも盤面再構築(`rebuildPhysics`、釘の追加・
  移動・削除・Undo・読込・リセットなどで呼ばれる)のたびに0へリセットされる
  (盤面を編集し直したら保留・変動状態もリセットするのが自然なため)。
- 左ツールバーの「ステータス」パネルに、保留(`保留数/SPEC.holdMax`)・
  状態(`phase` を「客待ち」「変動中」「リーチ中」「大当たり中」の日本語に
  変換)・モード(`mode` を「通常」「確変」「時短」の日本語に変換)・
  このセッションの総回転数・大当たり回数を表示する。

### 1000円回転数シミュレーション(賞球フィードバック込みの経済シミュレーション)

「簡易シミュレーション」(固定発射数でヘソ率から回転数を線形計算するだけ)
は、通常入賞で増えた玉を実際に撃ち込む効果(フィードバック)を反映しておらず、
「1000円で実際に何回転するか」を正確には測れない。これとは別に、
「1000円回転数シミュレーション」パネルを追加した。ロジックは
`src/main.ts` の経済モデル(`tickLaunch`/`handleBoardEvent` の賞球加算)を
参考に、使い捨ての `PhysicsCore`+`PachinkoGame` の組で再現している。

**設計判断の変遷(2回の修正を経て現在の形になった)**:

1. **1回目の実装**: 大当たり中のアタッカー賞球(13玉×最大9カウント×最大16R
   ≒最大1872玉)まで撃ち込み続ける仕様にしていたが、この機種の確率
   (1/99.9)では持ち玉を賞球込みで撃ち尽くすまでに大当たりを引く確率が
   約98%にも達し、ほぼ毎回このバイアスがかかってしまうことが実際の使用
   (ユーザー報告: 1000円で232回転)で発覚した。「釘配置の物理的な回転効率」
   ではなく「大当たりを引けたかどうかという運」を測ってしまっていた。
2. **2回目の実装**: `jackpot-start` が発生したら試行を打ち切り、複数回
   (60回)試行して平均を取る方式に変更したが、「60回は重い・遅い」という
   指摘を受けた。
3. **現在の実装(ユーザー提案)**: そもそも知りたいのは「通常時の回転効率」
   であり大当たりの運ではないため、**大当たり抽選そのものをシミュレーション
   中だけ無効化**すれば、複数試行して運をならす必要がなく、1回の実行だけで
   安定した値が得られる。`PachinkoGame` はコンストラクタで乱数関数を差し替え
   られる設計になっており(`src/game/stateMachine.ts`)、`src/game/lottery.ts`
   の `spin()` は変動1回につき最初に呼ぶ `rng()` が必ず大当たり判定
   (`hit = rng() < prob`)である。そこで、`PachinkoGame.update()` を呼ぶ
   直前に毎回「次の1回だけ大きい値(0.999999)を返す」フラグを立てる乱数
   ラッパーを注入することで、変動が始まるたびにその1回目の判定だけを
   確実にハズレにする。2回目以降の呼び出し(リーチ判定・図柄選択)は
   本物の `Math.random()` を使うため、図柄抽選内部の「左右不一致になる
   まで再抽選するループ」等が無限ループする心配もない。

現在のロジック:

- 持ち玉 `(1000 / SPEC.lendYen) * SPEC.lendBalls` = 250発から開始し、
  power で `SPEC.launchIntervalMs` 間隔で発射し続ける(実時間を待たず
  `SIM_STEP_MS=33ms` の固定刻みで一気に進める)。
- 発射のたびに持ち玉を1減らし、`heso`/`pocket` の通常入賞があれば
  `SPEC.payout` 分の玉を持ち玉に加算し、その玉も追加発射に使う(=要望通り
  「他入賞で増えた球数も含めての回転数」)。`denchu`/`attacker` は電サポ/
  大当たり中にのみ発生するが、大当たり抽選を無効化しているためそもそも
  発生しない。
- 万一 `jackpot-start` が来た場合(=大当たり無効化の前提が崩れた場合)は
  エラーとしてログに警告を出し、その時点で打ち切る。
- 安全装置として、発射数が20,000発、またはシミュレーション内時間が
  実時間換算30分(1,800,000ms)に達したら強制終了する。
- **参考値**: `DEFAULT_BOARD_DATA`・power=0.55/0.6/0.65 での実測(各1回
  実行)は回転数19〜24回転、発射数357〜413発。1回の実行(Bunでの検証で
  約5秒)だけで大当たりは一切発生せず、`scripts/simulate.ts` が示す
  「250発だけの単純換算で14〜19回転」よりやや高めに出るが、これは通常
  入賞の再投入分(発射数が250→360〜410前後まで伸びる)を正しく反映した
  差であり、大当たりの運は一切含まれない。

## ゲームロジック仕様 (エージェントB)

- 保留最大4。ヘソ入賞で保留+1(満タン時は無効、賞球のみ)。
- 変動時間: 通常時 基本8000ms / 保留3以上で4000ms / 電サポ中2000ms。
  リーチ時 +6000ms(ノーマル) or +14000ms(SP)。当たり時は70%がSP経由。
  ハズレ時のリーチ発生率: 通常時8% (うちSP 30%)。
- 図柄: 1..9 の3桁。確変当たり=奇数ゾロ目、通常当たり=偶数ゾロ目。
  16R確変は 7 揃い優先。ハズレはゾロ目にならない組合せ(リーチ時は左右一致・中±1)。
- 大当たり: ラウンドは attacker 入賞9個 or 9000ms で終了、インターバル1200ms。
- 通常当たり後: 時短100回。確変当たり後: 次回大当たりまで確変+電サポ。
- 時短中に当たった場合も振り分けは同一。時短を使い切ったら通常へ。
- 液晶(reels): DOM+CSSで3リール縦スクロール風。左→右→中の順に停止。
  リーチ時は中リールがスロー回転+「リーチ!」表示。大当たり中はラウンド数と
  「大当たり!」「確変突入!」等の演出表示。海テーマの配色。

## UI仕様 (エージェントC)

実際のパチンコホールにあるデータカウンター(呼び出しランプ)を忠実に再現する:

- 黒筐体 + 7セグ風表示。項目: 「大当り」(本日合計・赤)、「確変」(オレンジ)、
  「スタート」(現在回転数・緑)、「総スタート」、「前回」(前回当たりの回転数)。
- 大当たり履歴リスト(直近10件: 回転数/R数/確変か)。
- スランプグラフ(SlumpGraph): 黒背景に緑ライン(差玉プラス側)・赤(マイナス側)、
  0ライン・目盛り(±2500玉等)付き。パチンコ屋のデータ表示器風のデザインにする
  (一般的なチャート様式ではなく、実機データランプの様式を意図的に踏襲)。
- 台枠(frame.ts + CSS): 金色装飾枠・サイドランプ(大当たり中は虹色点滅)・上皿。
  上皿に持ち玉数(7セグ風)・投資額・「玉貸」ボタン(500円=125玉)・機種名プレート。
- ハンドル: 右下の円形ダイヤル。マウスドラッグ/タッチで回して開度0..1を調整、
  離しても保持(ストップボタンで0に)。キーボード←→でも微調整可。開度メーター表示。

## 音仕様 (エージェントD)

WebAudio のオシレーター/ノイズ合成のみで全SE・BGMを生成する(オリジナル曲。既存曲の模倣禁止):

- BGM: normal(のんびり海辺のマリンバ風ループ)、reach(緊張感のあるビート)、
  jackpot(明るいファンファーレ→アップテンポループ)、kakuhen(疾走感)、jitan(軽快)。
  8〜16小節のループをスケジューラで回す。切替はクロスフェード気味に。
- SE: types.ts の SeName 全種。nail は 80ms 以内の連打を間引く。
- ページ操作前は AudioContext を作らない(init() で解禁)。マスターゲインでミュート。

## 動作確認ループ

logger.ts の logger.log(category, message) を全モジュールの主要イベントで呼ぶこと
(発射・入賞・変動開始/停止・当たり・R進行・モード移行・UI操作・音切替)。
ユーザーは画面の「ログをダウンロード」ボタンでログを保存し、AIが解析する。
