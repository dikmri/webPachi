# webPachi 設計書

GitHub Pages で遊べる物理演算パチンコ台シミュレーター。
機種はオリジナルの海テーマ甘デジ「P ドリームオーシャン 99ver.」
(スペック・釘構成は一般的な羽根なしデジパチ/甘デジの典型構成を参考にしたオリジナル)。

## 技術スタック

- bun + Vite + TypeScript (strict) + matter-js (物理)
- 描画: Canvas 2D (盤面) + DOM/CSS (液晶・データカウンター・枠)
- 音: WebAudio API による手続き生成のみ (外部アセット・既存楽曲は一切使わない)
- デプロイ: GitHub Actions → GitHub Pages

## イベントフロー

```
[board] --BoardEvent--> [main] --転送--> [game(GameLogic)]
[game]  --GameEvent---> [main] --配線--> [reels] [audio] [dataCounter] [board(役物開閉)]
[frame] --handlePower-> [main] --launch--> [board]
[main] が PlayerState/Stats を所有し賞球加算・差玉サンプリングを行う
```

- 賞球テーブルは `SPEC.payout`。main が BoardEvent を見て持ち玉に加算する。
- `GameLogic.attackerShouldOpen` / `denSupport` を main が毎フレーム見て
  `board.setAttackerOpen()` / `setDenchuOpen()` を呼ぶ (電チューはゲート通過で1.5s開放、
  電サポ中のみ。開放制御は board 側でなく main 側タイマーでよい)。

## 盤面仕様 (エージェントA)

論理座標 480x660。重力は matter-js デフォルト方向(下向き)。

- 玉: 半径 5.5、restitution≒0.4、friction 小、密度デフォルト。上限 60個(超えたら古い玉を消す)。
  最大速度を保険として px/秒でクランプ(異常な跳ね返りによる速度増幅対策)。
- 釘: 半径 2.0 の静的円、restitution≒0.42。全体で約140本(縁釘38+千鳥格子51+天釘3+
  命釘2+袖釘6+寄せ釘16+道釘左右26+ジャンプ釘2)。
- 外枠: 左右壁・上部は円弧(中心 (240,330) 付近、半径≒310 の上半分)。
- 発射レール: 左端に幅約14の誘導路(外壁と内レールの間)。玉は左下 (18,640) 付近から
  power に応じた初速で上方へ発射され、円弧に沿って盤面上部へ入る。
  強いほど右奥(ブッコミ超え)まで飛ぶ。レール出口に戻り防止片。
- センター役物(液晶枠): 矩形 x:130..350, y:150..330 の静的ボディ。
  左横にワープ入口(通った玉はステージへ)。ステージ=液晶下の皿(y≒335)で玉が左右に転がり、
  中央から落ちるとヘソ上に落下(ヘソ入賞率高)。
- 風車(ふうしゃ): 左 (105,290)・右 (375,290)。回転する十字体(ピン留め)。玉を散らす。
- 縁釘(外周に沿った釘列): 外壁(上部円弧の左肩〜解放点〜右肩〜右辺)のすぐ内側
  (オフセット15px)に弧長26px間隔で一列に並ぶ。実機の「誘導レールのすぐ内側に
  ずっと並ぶ釘列」を再現し、強めのハンドルで右壁沿いを転がる右打ちルートの
  土台にもなる。機能釘(命釘・道釘・袖釘・寄せ釘等)に近すぎるものは自動的に間引く。
- 天釘: (216,110)(240,106)(264,110) の3本。レール解放点直下で玉を最初に振り分ける。
- 千鳥格子(上部〜中段の密な釘field): y:122〜334 の範囲に行間隔24px・列間隔26pxの
  スタッガード配置で機械的に生成(中央液晶・風車の可動域は除外)。実機らしい
  「規則正しいが経路で微妙に振り分けが変わる」密な釘盤を再現する。
- 袖釘(センター役物まわりの「ハの字」漏斗): 液晶枠の左下・右下のすぐ外側に
  各3本、上ほど役物に近く下へ行くほど外側へ開く配置。抜けた玉を下段の
  寄せ釘へ引き継ぐ。
- 寄せ釘: 袖釘の下、y:408〜510 に左右各8本。下へ行くほど中央(ヘソ)側へ収束し、
  道釘・ジャンプ釘・命釘へ滑らかに繋がる「漏斗」を構成する。
- 道釘: y≒546〜566 に左右対称の水平列(間隔約13〜14、玉が上を転がる)。
  左列 x:40〜224・右列 x:256〜440。各列途中1箇所こぼしの切れ目。
  レール解放点が power で広く散らばるため、右側にも同型の道釘列を用意して
  片側だけに回転率が偏らないようにしている。
- ジャンプ釘→命釘: 道釘の終端で玉を跳ねさせ、命釘2本(中心間隔≒70)の間へ。
- ヘソ(スタートチャッカー): (240,558) 幅11のセンサー。
- 電チュー: ヘソ直下 (240,588)。閉時は入賞不可、開時は幅22の受け皿。
- スルーゲート: 左道中 (100,340)。通過検知のみ。
- アタッカー: (240,612) 幅56の開閉フラップ。開時は落下玉の大半を吸い込む。
- 一般入賞口: 左下に3個 (60,600)(85,614)(110,624) 付近。
- アウト口: 最下部 (240,648)。入賞しなかった玉を回収。
- 盤面背景: 海テーマ(青グラデ・泡・光条・砂地)を Canvas で手続き描画。「それっぽく」豪華に。

**貫通(tunneling)対策**: 固定ステップを 1000/240 秒に細分化、
positionIterations/velocityIterations を既定値より引き上げ、玉の最大速度を
保険としてクランプした上で、玉と釘専用の簡易連続衝突判定(CCDスイープ)を実装。
Matter.Engine.update() 前後の玉の座標を線分とみなし、近傍の釘(空間分割で
絞り込み)に対して「線分と釘中心の最短距離」が当たり判定幅を下回っていないか
幾何学的に判定し、両端点が当たり判定の外側なのに軌跡だけが貫通している場合
(=matter-js の離散衝突では検出されない tunneling)のみ接触点まで押し戻して
反射させる。scripts/simulate.ts に検証用カウンタ(検出・補正回数/補正漏れ回数)
を実装し、強めのpower(0.7〜1.0)で数千発のストレステストを行い
補正漏れ0件を確認済み。

**調整目標(scripts/simulate.ts で検証)**: 適正ハンドル(power 0.55〜0.65 付近)で
1000発打って ヘソ入賞 55〜75発 (千円=250発あたり14〜19回転)。
シミュレーションは matter-js を固定タイムステップで回し、入賞内訳を表で出力する。
(シミュレーションは発射ごとに乱数ジッターを与えているため実行のたびに数発程度の
ばらつきがある。目安から外れる回が稀にあっても、複数回実行して大勢として
55〜75発付近に収まっていれば正常。)

## 盤面エディタ (エージェントA)

盤面の釘・役物配置を「人力で」調整したいという要望に応え、ブラウザ上で
座標を直接編集できるツールを用意した。`editor.html` を開くと使える
(`index.html` とは完全に独立した別ページ)。

### 「筐体」と「取付部品」の切り分け

実機になぞらえ、盤面を2種類の要素に分けている。

- **筐体(編集不可・固定)**: 外枠レール、センター役物(液晶枠)の形、
  ステージ、ワープ、アウト口。これらは `src/board/layout.ts` に定数として
  残り、エディタでは編集できない。
- **取付部品(編集可能)**: 釘・ヘソ・電チュー・スルーゲート・アタッカー・
  一般入賞口・風車の**座標**。実機の技術者が釘配置やチューリップ位置を
  ハンマーで調整するのと同じ発想で、これらは `BoardData` という単純な
  データ型として外部化し(`src/board/boardData.ts`)、エディタから自由に
  編集・保存できるようにした。釘の個別半径やヘソ等の半幅も編集できるが、
  釘の既定半径(NAIL_RADIUS)・一般入賞口の半幅(POCKET_HALF_WIDTH)・
  風車の腕の寸法や回転速度など「サイズ・挙動」は引き続き固定。

物理演算(`PhysicsCore`)・描画(`drawBoard`)はどちらも `BoardData` を
引数として受け取るように統一されているため、本編(`src/main.ts`)は
`new Board(data)` に読み込んだ `BoardData` を渡すだけでよく、エディタで
組んだ盤面がそのまま実機と同じ物理エンジンで動く(挙動の二重実装がない)。

### データの保存の仕組み

- 編集するたびに(デバウンスして自動保存)、`localStorage` の
  キー `webpachi-board-data-v1`(`BOARD_DATA_STORAGE_KEY` として
  `src/board/boardData.ts` がエクスポート)へ現在の `BoardData` を
  JSON 保存する。
- 本編(`src/main.ts`)は起動時にこのキーを読み、`isValidBoardData()` で
  検証した上で問題なければその盤面を、無ければ `DEFAULT_BOARD_DATA`
  (従来通りの盤面)を使う。**つまりエディタで調整した盤面は、保存すると
  そのまま本編のプレイ画面にも反映される。**
- 「JSONをエクスポート」で `.json` ファイルとしてダウンロードでき、
  「JSONを読み込む」でファイルから復元できる(どちらも
  `isValidBoardData()` で形状を検証し、不正なら画面にエラー表示する)。
- 「デフォルトに戻す」で `DEFAULT_BOARD_DATA` のコピーに戻せる
  (localStorage も上書きされる)。

### エディタの操作方法

- **釘の追加**: キャンバスの空いている場所をクリック。
- **釘・役物の移動**: 既存の釘・役物(ヘソ/電チュー/ゲート/アタッカー/
  一般入賞口/風車)をドラッグ。役物は移動のみ可能で削除はできない。
  釘は選択して `Delete`/`Backspace` キーで削除できる。
- 選択中のアイテムは右側ではなく左側の「プロパティ」パネルに表示され、
  x/y(・釘なら半径、役物なら半幅)を数値入力でも編集できる。
- 「表示切り替え」パネルのトグルで、
  - 釘の物理半径(実際の釘の太さ)の円
  - 釘の実効半径(釘半径+玉半径 `BALL_RADIUS`)の破線円 ―
    **2本の釘の間を玉がすり抜けられるかどうかの実際の目安**であり
    パチンコ盤面設計上もっとも重要な可視化(デフォルトON)
  - テスト発射中の玉自身の当たり判定円
  - グリッドスナップ(2px単位)
  を表示・切り替えできる。
- 「テスト発射」パネルの power スライダーと「1発発射」ボタン、または
  「オート連射」チェックボックス(ハンドルを握りっぱなしにした状態を再現)で、
  現在編集中の盤面がそのまま `PhysicsCore` に反映された状態で実際に玉を
  転がして確認できる(編集のたびに物理ワールドを再構築するため、
  常に最新の釘配置で動く)。
- 「簡易シミュレーション」パネルで power と発射数を指定して「実行」を押すと、
  `scripts/simulate.ts` と同じロジックをブラウザ内で(専用の使い捨て
  `PhysicsCore` を使って)ヘッドレス実行し、ヘソ/電チュー/ゲート/一般入賞/
  アタッカー/アウトの内訳とヘソ率・回転数の目安をその場に数値表示する。
  CLI を使わずにブラウザだけで回転率チューニングが完結する。
- 右側の「動作ログ」パネルには操作(釘追加・移動・削除・保存・読込・
  シミュレーション結果等)が逐次記録され、「ログをダウンロード」で
  テキストファイルとして保存できる。
- `Ctrl+Z` または「元に戻す」ボタンで直前の編集を1手戻せる(Undo、最大30手)。

### CLIからの検証

エディタでJSONエクスポートした盤面は、CLIのヘッドレスシミュレーターでも
そのまま検証できる。

```
bun run simulate                    → DEFAULT_BOARD_DATA を使う(従来通り)
bun run simulate -- ./my-board.json → エクスポートしたJSONの盤面で検証する
```

## ゲームロジック仕様 (エージェントB)

- 保留最大4。ヘソ入賞で保留+1(満タン時は無効、賞球のみ)。
- 変動時間: 通常時 基本8000ms / 保留3以上で4000ms / 電サポ中2000ms。
  リーチ時 +6000ms(ノーマル) or +14000ms(SP)。当たり時は70%がSP経由。
  ハズレ時のリーチ発生率: 通常時8% (うちSP 30%)。
- 図柄: 1..9 の3桁。確変当たり=奇数ゾロ目、通常当たり=偶数ゾロ目。
  16R確変は 7 揃い優先。ハズレはゾロ目にならない組合せ(リーチ時は左右一致・中±1)。
- 大当たり: ラウンドは attacker 入賞9個 or 9000ms で終了、インターバル1200ms。
- 通常当たり後: 時短100回。確変当たり後: 次回大当たりまで確変+電サポ。
- 時短中に当たった場合も振り分けは同一。時短を使い切ったら通常へ。
- 液晶(reels): DOM+CSSで3リール縦スクロール風。左→右→中の順に停止。
  リーチ時は中リールがスロー回転+「リーチ!」表示。大当たり中はラウンド数と
  「大当たり!」「確変突入!」等の演出表示。海テーマの配色。

## UI仕様 (エージェントC)

実際のパチンコホールにあるデータカウンター(呼び出しランプ)を忠実に再現する:

- 黒筐体 + 7セグ風表示。項目: 「大当り」(本日合計・赤)、「確変」(オレンジ)、
  「スタート」(現在回転数・緑)、「総スタート」、「前回」(前回当たりの回転数)。
- 大当たり履歴リスト(直近10件: 回転数/R数/確変か)。
- スランプグラフ(SlumpGraph): 黒背景に緑ライン(差玉プラス側)・赤(マイナス側)、
  0ライン・目盛り(±2500玉等)付き。パチンコ屋のデータ表示器風のデザインにする
  (一般的なチャート様式ではなく、実機データランプの様式を意図的に踏襲)。
- 台枠(frame.ts + CSS): 金色装飾枠・サイドランプ(大当たり中は虹色点滅)・上皿。
  上皿に持ち玉数(7セグ風)・投資額・「玉貸」ボタン(500円=125玉)・機種名プレート。
- ハンドル: 右下の円形ダイヤル。マウスドラッグ/タッチで回して開度0..1を調整、
  離しても保持(ストップボタンで0に)。キーボード←→でも微調整可。開度メーター表示。

## 音仕様 (エージェントD)

WebAudio のオシレーター/ノイズ合成のみで全SE・BGMを生成する(オリジナル曲。既存曲の模倣禁止):

- BGM: normal(のんびり海辺のマリンバ風ループ)、reach(緊張感のあるビート)、
  jackpot(明るいファンファーレ→アップテンポループ)、kakuhen(疾走感)、jitan(軽快)。
  8〜16小節のループをスケジューラで回す。切替はクロスフェード気味に。
- SE: types.ts の SeName 全種。nail は 80ms 以内の連打を間引く。
- ページ操作前は AudioContext を作らない(init() で解禁)。マスターゲインでミュート。

## 動作確認ループ

logger.ts の logger.log(category, message) を全モジュールの主要イベントで呼ぶこと
(発射・入賞・変動開始/停止・当たり・R進行・モード移行・UI操作・音切替)。
ユーザーは画面の「ログをダウンロード」ボタンでログを保存し、AIが解析する。
